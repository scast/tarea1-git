Tarea 1

Esta primera tarea consiste en familiarizarse un poco con herramientas que deben ser comunes para el desarrollo de cualquier proyecto. En las siguientes instrucciones posiblemente hayan algunos términos desconocidos si nunca has trabajado con Git; queda de tu parte investigar lo que significan para poder realizar la actividad.

Como parte práctica de esta tarea, tendrás que seguir los siguientes pasos:

  1. Crear una cuenta en github.
  2. Realizar un fork en github de este repo: http://github/devlabs-usb/tarea-1.
  3. Clonar tu propio fork usando git clone.
  4. Hacer merge del branch conflict-branch que existirá en tu repositorio a tu branch gh-pages (ver git help checkout y git help merge).
  5. Resolver los "merge conflicts" que surgirán al hacer el merge.
  6. En tu propio repositorio, rellena tu archivo "index.html" con las respuestas a las preguntas teóricas que se encuentran acá.
  7. Hacer push de los cambios a "index.html" a tu repositorio cuando termines de rellenar las respuestas.
  8. Enviarnos un correo a devlabs-usb@talpor.com con el URL de tu repositorio en GitHub, tu nombre y carnet. Bonus: Si entiendes como funcionan los git pages, mándanos el link a tu página de respuestas.

3. En un contexto de la herramienta de control de versiones Git: definir 'working directory', 'index', 'local repo', 'remote repo', 'dirty working tree' y 'merge conflict'.

Working directory
Básicamente es donde se encuentra la carpeta .git. Es la carpeta que obtienes despues del git clone o del git init.
Index
Donde se van agregando los cambios que se van a guardar (commit). Es el staging area previo al commit.
Local Repo
Es tu copia local de un repositorio. Es totalmente independiente de cualquier otro repositorio.
Remote Repo
Es un repositorio git remoto, para alguna definición de remoto que dependiendo del caso podría ser alguna de: un repositorio git en otra carpeta en mi misma computadora, en otra computadora en mi misma red, un repositorio en la India, cualquier repositorio git no igual a mi local repo. Ver git remote.
Dirty Working Tree
Es cuando existen cambios locales que no han sido commiteados.
Merge conflict
Es lo que ocurre cuando git no puede automáticamente realizar el merge de un archivo proveniente de dos branches diferentes. Ejemplo: al hacer merge de un feature branch sobre master puede que haya archivos donde git no sea capaz de realizar el merge automáticamente y necesite ayuda para decidir que hacer. Esto suele ocurrir cuando el desarrollo sobre master siguió su rumbo y se realizaron cambios en alguno de los mismos archivos que el feature branch está tratando de merge. Ver pregunta 1.

Por ejemplo, el comando git commit, lee el indice, construye un commit, lo agrega al repositorio actual y finalmente vacía el indice.

git diff git fetch git add git commit git pull git push git checkout
working dir R W W
index R* W R/W W*

* denota que dependiendo de los parámetros el comando puede leer/escribir de alguna de las filas demarcadas con el asterisco. Por ejemplo, git diff solo muestra diferencias en los archivos del working directory que no están en el index pero con un parámetro muestra las diferencias del workspace con el commit especificado.
